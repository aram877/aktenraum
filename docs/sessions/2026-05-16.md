## Multi-agent audit (input)

Four agents ran in parallel against the repo before any code was written. Their reports framed the rest of the session:

1. **Architecture** — verdict: "well above average for a personal project being productized." No critical issues. Two important gaps: hardcoded `~/aktenraum/` paths (violates ADR-002 #2), and no SIGTERM graceful shutdown.
2. **Performance** — single dominant problem: `PaperlessClient` has zero caching, so the poller / extractor / propagator / indexer all issue 3-10 redundant HTTP round-trips per doc. The `aktenraum-api`'s `PaperlessGateway` already implements the right TTL pattern; lifting it into core is the single biggest win.
3. **Security** — `SameSite=Lax` already blocks most CSRF in default config; still surfaced three concrete must-fix items: no second-line CSRF check, non-timing-safe webhook secret comparison, `COOKIE_SECURE=false` shipped as default.
4. **Code quality** — TOCTOU on `swap_lifecycle_tag` (concurrent approve loses concurrent propagation's tag adds), `_get_custom_field_ids` cached forever (so a `bootstrap-paperless.sh` run during a live container needs a restart to be visible), `_extract_doc_id` parses doc id out of FAILURE rows too.

User authorised the full close-out without further questions. 22 tracked tasks; all completed.

## What shipped

### Security

- **CSRF defence-in-depth** (new file `services/aktenraum-api/src/aktenraum_api/middleware.py`). Two middlewares wired in `main.py`:
  - `CSRFMiddleware` rejects state-changing requests (`POST`/`PUT`/`PATCH`/`DELETE`) and the sensitive GETs (`/preview`, `/download`) when `Sec-Fetch-Site` is `cross-site`. Internal callers carrying `X-Aktenraum-Secret` bypass. Documented as **ADR-003**.
  - `SecurityHeadersMiddleware` sets `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy: no-referrer` on every response.
- **`hmac.compare_digest`** replaces direct string equality in three webhook-secret comparisons: auto-tagger webhook (`services/auto-tagger/src/auto_tagger/webhook.py`), `/api/settings/active-llm-model`, `/api/documents/{id}/type-fields`. Timing-safe.
- **`COOKIE_SECURE` defaults to `True`** in `aktenraum-api/config.py`. The localhost dev env file (`docker/aktenraum-api.env`) explicitly opts out to `false` so http://localhost:8080 keeps working; production deploys (Tailscale + HTTPS, public hosting) inherit the secure default.
- **Hard gate restored on auto-approve.** `auto_approve_types` is no longer deprecated — auto-approve requires **both** `confidence ≥ AUTO_APPROVE_CONFIDENCE` AND `document_type ∈ AUTO_APPROVE_TYPES`. Empty list (the new default) disables auto-approve entirely. Prompt-injection that emits `confidence=0.99` can no longer skip the human review queue. Test matrix in `services/auto-tagger/tests/test_tagger.py` rewritten to cover both gates.
- **Upload caps + nginx hardening**:
  - `services/aktenraum-api/src/aktenraum_api/config.py` gains `upload_max_file_bytes` (25 MB default) and `upload_max_files_per_request` (20). Per-file MIME allowlist in `documents/router.py` covers PDF, common images, MS/OpenOffice formats, plain text — anything else returns "Unsupported content-type" for that file (other files in the batch still process).
  - `docker/nginx/nginx.conf`: `client_max_body_size 500m`, `server_tokens off`, `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: no-referrer`, full CSP (`default-src 'self'; …; frame-ancestors 'none'`).

### Bugs

- **`PaperlessGateway.swap_lifecycle_tag` TOCTOU fixed.** Now reads → plans → PATCHes → **re-reads to verify**; if a concurrent writer interleaved (set difference), replays the swap on the fresh state up to 3 attempts before raising the new `PaperlessConflictError`. The error maps to HTTP 409 on the inbox and documents routers with a SPA-facing message that tells the user to refresh. Closes the "two concurrent approvals silently drop `ai-propagated`" failure mode.
- **`PaperlessClient._get_or_create_named` race-tolerant.** On POST 400 (unique-name constraint trip from a parallel writer creating the same correspondent / tag / document_type), re-GETs once and reuses the winner's id instead of bubbling up an error that would tag the doc `ai-propagation-error`. Logs at info level when a race is observed.
- **Custom-field cache invalidation.** Both `PaperlessClient._get_custom_field_ids()` and `PaperlessGateway._get_custom_field_ids()` are now TTL-cached (5 min) instead of cached-for-process-lifetime. The gateway's `patch_document_custom_fields` performs a one-shot cache refresh on an unknown field name before logging `paperless_unknown_custom_field` — so running `bootstrap-paperless.sh` against a live container is visible within one PATCH instead of needing a restart.
- **`/api/documents/task/{uuid}` doc-id extraction.** `_extract_doc_id` now only regex-parses `result` strings on `SUCCESS` task rows. FAILURE messages like "Failed because document id 7 was a duplicate" previously pointed the SPA's upload poller at the wrong doc until the 120s ceiling hit. Test added.
- **Webhook `QueueFull` handled.** `auto_tagger.webhook.trigger_extraction` catches `asyncio.QueueFull` from `put_nowait`, logs `webhook_queue_full`, and returns 503 instead of 500. Mass-import scenarios (paperless restoring backed-up docs) used to silently 500 the webhook handler.
- **SSE error body surfaced.** `apps/web/src/lib/ai.ts`'s `streamAsk` reads `resp.text()` / `{detail: ...}` on `!resp.ok` instead of returning a bare `502 Bad Gateway`. The user now sees the actual reason (`Paperless rejected the API token`, validation message, …).

### Performance

- **TTL caches in `PaperlessClient`** (`packages/aktenraum-core/src/aktenraum_core/paperless/client.py`). Five-minute default TTL on `_get_tag_id`, `_get_custom_field_ids`, `get_entity_name_map`. Create paths (`_get_or_create_named`, `ensure_tag`, `ensure_custom_field`) invalidate their own keys so freshly-created entities are visible on the very next access. Public `invalidate_caches()` for callers that mutated Paperless out-of-band. **Single biggest backend win** — drops the auto-tagger's Paperless traffic by roughly 10× across poll/extract/propagate paths.
- **`/api/documents/in-flight` count memoised** for 15 seconds at module scope. Lock-guarded against the thundering-herd cold start. 100 open SPAs no longer translate to 200 expensive `tags__id__in` list calls per minute.
- **Reranker pre-warmed in lifespan startup.** `aktenraum-api/main.py` schedules `LocalReranker._ensure_loaded()` as a background task during lifespan startup; the API returns immediately to serve requests, and the first `/ask` no longer pays the ~5-minute HuggingFace-download cliff inside a request handler.
- **Bulk-approve throttled to 4 concurrent POSTs** (`apps/web/src/lib/inbox.ts`). Pure-TS concurrency limiter (`_runWithConcurrency`); same shape ready to lift into `useBulkReprocess` if needed.
- **`useProcessingState` polling** dropped from 2s → 5s with matching `staleTime`. Cuts ~60% of the per-tab steady-state request load.
- **`patch_document_custom_fields` accepts `prefetched_doc`.** Callers that already hold a fresh doc dict (inbox approve / edit, library field-patch) pass it through and skip the gateway's internal merge-read. Saves one GET per field-edit.
- **Route-level code splitting in the SPA** (`apps/web/src/router.tsx`). Every route is `React.lazy`-imported behind a single `RouteSuspense` wrapper. Build output goes from a monolithic bundle to a 357 KB main + per-route 2-17 KB chunks (Library 17 KB, LibraryReview 11 KB, InboxReview 10 KB, Upload 7 KB, Ask 4 KB, Find 4 KB, Settings 5 KB, etc.).

### Reliability

- **Graceful SIGTERM handling in the auto-tagger.** `run()` wires `loop.add_signal_handler(SIGTERM/SIGINT, …)` to flip an `asyncio.Event`, then races the `gather` against the shutdown wait — when shutdown fires, cancels the gather cleanly so each loop finishes its current iteration before the process exits.
- **Propagator's lifecycle-flipping PATCH shielded** with `asyncio.shield()`. Cancellation can't tear it apart mid-swap; the doc either ends up correctly transitioned to `ai-propagated` or stays in `ai-approved` for the next pass. No more "ai-approved cleared but ai-propagated not added" mid-shutdown state.

### Desktop-app constraint (ADR-002 #2)

- **`${AKTENRAUM_DATA_DIR:-${HOME}/aktenraum}` substitution** on all 10 compose bind mounts in `docker/docker-compose.yml`. Documented in `docker/.env.example` with the platform-appropriate defaults Tauri will set. Localhost dev keeps working with zero config because the default falls back to `~/aktenraum`.

### Doc + drift fixes

- **CLAUDE.md** — `20 types` → `26 types` in two places; `chunk*` → "repeated `chunk`" event description; live with ADR-003 link.
- **`Bußgeldbescheid` spelling** unified in `services/auto-tagger/src/auto_tagger/tagger.py` `SYSTEM_PROMPT`. Both the type taxonomy and the per-type title template now use the ß form (was `Bussgeldbescheid` in two definitions, `Bußgeldbescheid` elsewhere — confused small local LLMs).

## Follow-up commit: small-LLM field-drop fixes + workflow walkthrough

User reported `summary_de` arriving empty from the small local model and noted the same drop on `reference_numbers` and `suggested_tags`. Investigation confirmed the same root cause as the previous `ai_title` / `confidence_reason` fixes: every field in `DocumentExtraction` with a Pydantic default (empty string or empty list) is treated as optional by the LLM, so small models routinely omit them and Pydantic silently accepts the default. The previous fix landed synthesizers for `ai_title` and `confidence_reason` but never extended to the three remaining fields.

Shipped:

- **`_synthesize_summary_de`** in `services/auto-tagger/src/auto_tagger/tagger.py` — deterministic German summary built from `document_type` + `correspondent` + `key_dates.issue` + `ai_title` + `reference_numbers` / `suggested_tags`. Never empty (since `document_type` is required on the model). Mirrors the existing `_synthesize_ai_title` pattern.
- **`_extract_reference_numbers_from_text`** in the same file — regex sweep over the full OCR `content` for nine common German labels (Aktenzeichen, Az., Rechnungsnr., Vertragsnr., Kundennr., Vorgangsnr., Bestellnr., Auftragsnr., Policennr., Steuernr.). Conservative: requires the labelled prefix, captures 4-32-char alphanumeric runs only, dedups case-insensitively, returns at most 5 values in label-priority order. Operates on full `content` (not the LLM-truncated `text`) so a ref number on page 6 of a long doc still surfaces.
- Both wired into `process_document` between the existing `confidence_reason` fallback and the Paperless PATCH. Trigger conditions: each fires only when the LLM emitted empty for that specific field. The LLM's value wins whenever it bothered to emit one.
- Prompt rules tightened from one-line "summary_de muss genau 3 Sätze auf Deutsch enthalten" to explicit per-field multi-line rules with concrete examples (per-sentence shape for the summary, label list + capture target for reference_numbers, "no doc-type repetition, 2-5 content terms" with two worked examples for suggested_tags).
- `suggested_tags` deliberately gets prompt-only treatment — auto-synthesis would risk fabricating useless tags (year-as-tag, doc-type-as-tag) that the propagator would then materialise as real Paperless tags.

Tests: 18 new tests in `services/auto-tagger/tests/test_tagger.py` covering both helpers across full / minimal / empty extractions, label-priority order, dedup case-insensitivity, value-too-short / value-too-long rejection, and the limit cap. Full suite: 475 passed (was 457).

## Follow-up commit: modular AI prompts

Implements `docs/plans/modular-ai-prompts.md`. Static one-size-fits-all prompts were causing two failure shapes: the filter LLM ignored salary/tax/insurance-binding verbs because nothing in the few-shots demonstrated those mappings, and the answer LLM saw irrelevant Pass / Stromrechnung / Wizz-Air examples regardless of which doc types were actually retrieved. Both prompts now assemble per-type pieces at request time.

Adjusted the original draft before implementing: `22 doc types` → `26` (the codebase has 26); field labels resolved at render time from `TYPE_FIELD_SCHEMA` instead of being duplicated in the module table (so adding a typespecific field in one place ripples everywhere); intent detection only feeds the filter prompt (the answer prompt has candidates and can read their doc types directly — more reliable than keyword matching).

Shipped:

- **`services/aktenraum-api/src/aktenraum_api/ai/intent.py`** — pure keyword classifier returning a set of `Intent` values (`SALARY`, `SPENDING`, `TAX`, `INSURANCE`, `HOUSING`, `MEDICAL`, `ID_DOCUMENT`, `CAR`, `CONTRACT`). Substring match by default (handles German compounds: "Mietvertrag" → vertrag, "Steuererstattung" → steuer); short ambiguous keywords (`pass`, `lohn`) opt into strict whole-word matching via `_STRICT_KEYWORDS` to avoid colliding with "Passwort" / "lohnt". `INTENT_DOC_TYPES` maps each intent to the doc types it implies, used by `doc_types_for_intents()`.
- **`services/aktenraum-api/src/aktenraum_api/ai/prompt_modules.py`** — one `DocTypeModule` per `DocumentType` (frozen dataclass: `filter_examples`, `answer_hint`, `answer_example`). Import-time `_assert_all_doc_types_covered()` rejects any new enum value that lacks a module so the table cannot drift. `field_labels_for(doc_type)` reads directly from `TYPE_FIELD_SCHEMA`; `parse_document_type()` is the defensive enum lookup for candidate dicts.
- **`ai/prompt.py`** — `build_messages` threads `query` into `_build_system_prompt`; after the six static few-shots the assembler calls `detect_intents(query)` → `doc_types_for_intents()` → unions the matched modules' `filter_examples` (dedup by question text). No-intent questions fall through to the static set unchanged → prompt-cache hits for the common browse case stay intact.
- **`ai/answer_prompt.py`** — both `build_answer_messages` (JSON envelope) and `build_streaming_answer_messages` (SSE) now compose:
  - **System "Feld-Hinweise"** — `_assembled_field_hints(candidates)` enumerates each distinct doc type present, prints its field labels (from `TYPE_FIELD_SCHEMA`) and the module's `answer_hint`. Falls back to one generic "use typespecific fields" line when no candidate matches a known module (empty / all-`Sonstiges`).
  - **User "Beispiele"** — `_assembled_examples(candidates, json_mode=...)` emits one module example per distinct doc type. The streaming path uses the `[Quelle: N]` form verbatim; the JSON path wraps the same string in `{"answer_de": ..., "cited_ids": [...]}` via `_to_json_envelope` so the two paths cannot drift. Plus the always-on `_STATIC_AGGREGATION_EXAMPLE` (Wizz-Air style) since cross-doc aggregation is a citation-format concern, not domain.
  - Removed the four hardcoded Pass / Stromrechnung / Wizz-Air / verdient examples and the hardcoded Feld-Hinweise lines that mentioned Ausstellung + Geldbeträge globally — replaced by candidate-driven assembly.

Tests: 41 new tests across `test_ai_intent.py` (15), `test_ai_prompt_modules.py` (11), and extensions to `test_ai_prompt.py` (+3) and `test_ai_answer.py` (+4). The integrity tests assert every `DocumentType` enum value has a module and that field labels round-trip through `TYPE_FIELD_SCHEMA`, so future schema additions cannot leave the prompt half-updated. One existing assertion (`test_answer_prompt_omits_type_section_when_empty`) was tightened from `"Typenspezifische Felder" not in user` to `"\n  Typenspezifische Felder:" not in user` because the *example* (no indent) is now allowed in the user message even when a candidate's `type_specific_fields` is empty — the guard now precisely targets the indented candidate-render path. Full suite: 512 passed (was 475). Ruff clean.

`docs/plans/modular-ai-prompts.md` updated in place to capture the adjustments + final status; the original draft was overwritten because we are pre-merge and the plan is the spec, not a historical record.

Also shipped: **`docs/workflow.md`** — a plain-English "follow one document through the system" walkthrough (cast of services → upload journey → AI extraction → user approval → AI Q&A → background loops → who's responsible for what → lifecycle tag table → what-happens-when-things-go-wrong). ## Follow-up commit: Ask AI — three-bug fix for query failures

User reported that every Ask AI query returned "Ich konnte das in den Dokumenten nicht finden." regardless of question. Root-cause investigation identified three independent bugs acting together.

### Bug 1 — February (and end-of-month) dates cause silent 3× retry + error

`qwen2.5vl:7b` consistently generates `date_to: "2026-02-29"` for February queries. 2026 is not a leap year; Pydantic's `date` type rejects the impossible day, the `OllamaBackend` retries three times with identical output, and `_stream_answer_events` surfaces "Filter konnte nicht extrahiert werden." The same pattern hits any month whose last day the model guesses wrong (Nov 31, etc.).

**Fix**: `field_validator("date_from", "date_to", mode="before")` in `SearchFilter` (`services/aktenraum-api/src/aktenraum_api/ai/schemas.py`). Parses ISO strings, detects day > `calendar.monthrange(year, month)[1]`, clamps to the real last day. Valid dates and non-string inputs pass through unchanged. 4 new tests in `test_ai_router.py`.

### Bug 2 — `OLLAMA_ANSWER_MODEL` env var silently overrides the UI quality picker

`docker/aktenraum-api.env` had `OLLAMA_ANSWER_MODEL=qwen2.5vl:7b`. `deps._build_backend` checks `settings.ollama_answer_model` first and short-circuits before reading `settings_service.get_active_answer_model(session)`. The Settings UI's "high" quality (= `gemma4:26b`) was never reached for the answer step. The user's attempts to switch models in the UI had zero effect.

**Fix**: Cleared `OLLAMA_ANSWER_MODEL=` in the env file. The settings service now controls the answer model; DB has `answer_llm_quality = 'high'` which maps to `gemma4:26b` in `quality.py`.

### Bug 3 — LLM asked to sum 12 salary figures produces "nicht gefunden"

With 12 Gehaltsabrechnung candidates (annual query like "wie viel habe ich in 2023 verdient"), the model received all 12 EUR-formatted bruttogehalt/nettogehalt values and was expected to add them up in its head. Both `qwen2.5vl:7b` (7B) and `gemma4:26b` (26B) gave up and emitted the "nicht gefunden" fallback instead.

**Fix**: `_compute_type_aggregations(candidates)` in `answer_prompt.py` pre-computes sums of all monetary (`EUR`-prefixed) `type_specific_fields` grouped by doc type, when ≥2 candidates share the same type. A "Berechnete Summen" block is injected between the question and "Verfügbare Dokumente:" in both the streaming and JSON-envelope user prompts. The model reads a pre-computed total (`EUR45400.00`) instead of doing arithmetic. 3 new tests in `test_ai_answer.py`.

**Verified by user on real data.** Question "wie viel habe ich in 2023 verdient" → "Im Jahr 2023 hast du ein Bruttogehalt von 45.400,00 € und ein Nettogehalt von 29.438,00 € verdient." (12 payslips, correct aggregate).

Full suite: 519 passed (was 512). Ruff clean.
