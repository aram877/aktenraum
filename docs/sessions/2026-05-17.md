## What shipped

User-reported failure mode: `/ask` returns "Ich konnte das in den Dokumenten nicht finden." in the prose while the SPA still renders the retrieved candidate documents as source cards underneath. Looks like the AI is lying about its own search.

Root cause traced to `services/aktenraum-api/src/aktenraum_api/ai/router.py`:

- Both the JSON `/answer` (line ~200) and the streaming `/answer/stream` (line ~497) branches contain a "if the model wrote prose but emitted zero `[Quelle: N]` markers, back-fill citations from the retrieval set" rule. Intent: prose without a verifiable source is useless. Side-effect: when the prose IS the prompt-baked denial template ("Ich konnte das in den Dokumenten nicht finden."), the back-fill triggers and the SPA shows source cards beneath a "not found" message.

Fix:

- New `_DENIAL_RE` regex + `_is_denial_answer(text)` helper next to the existing `_is_degenerate_answer`. Detects the prompt-baked template plus four common small-model variants ("nicht in den Dokumenten", "keine passenden Dokumente", "keines der Dokumente enthält/nennt/beantwortet"). Length-capped at 200 chars so partial answers like "Du hast bei X gebucht. Die Preise konnte ich nicht finden, aber die Buchungen sind verlinkt." are NOT treated as denials — they carry real content and keep their citations.
- Both back-fill branches now gate on `not _is_denial_answer(answer_text)`. Denial prose → `citations = []`. Real prose → back-fill as before.

Tests added in `services/aktenraum-api/tests/test_ai_answer.py`:

- Five unit tests for `_is_denial_answer` (template hit, common variants, real-prose negatives, empty input, long-partial negative).
- One e2e test for JSON `/answer` denial path.
- One e2e test for streaming `/answer/stream` denial path.

Total: 519 passed (was 512), ruff clean.

CLAUDE.md updated with one row in the `/ask` description noting the new behaviour so future sessions don't reintroduce the back-fill.

## Third pass: web-trash-management (OpenSpec change)

Discovered while debugging the user's "delete also from Paperless trash" request: the existing `paperless_gw.delete_document` had a docstring claiming hard-delete ("Paperless does not soft-delete…") that was false. Paperless 2.x always soft-deletes — every doc the user ever clicked Löschen on has been sitting in `/api/trash/` recoverable for 30 days, then auto-purged. The SPA never surfaced any of that. Same investigation surfaced an orphaned-Qdrant-chunks bug — soft-delete never touched the vector store, so RAG could still answer from "deleted" docs.

OpenSpec change: `openspec/changes/web-trash-management/` (proposal, design, two delta specs, tasks). Design picked the two-step Trash → empty model over one-shot hard-delete (per user) and bundled the Qdrant cleanup with the hard-delete path.

**What landed:**

## Fifth pass: duplicate-flagging (OpenSpec change)

Picked up the duplicate-doc-flagging feature postponed from the library work. User narrowed scope to the tightest viable v1:

- **Detection signal**: field-based only. Same correspondent + same `ai_issue_date` AND (monetary amounts within 0.01 EUR OR shared `ai_reference_numbers` entry). No text similarity, no embeddings.
- **Trigger**: auto on every successful propagation. No backfill script, no manual command.
- **User actions**: flag-only — no new SPA route, no compare-and-resolve UI, no dismissal store. User filters Library by `?tags=ai-duplicate` (existing tag-filter URL state) and resolves via the existing Löschen → Papierkorb flow.

**What landed:**

1. **New `ai-duplicate` auxiliary tag** registered in `bootstrap-paperless.sh` with color `#a855f7` (purple, distinct from the lifecycle palette). Documented in `aktenraum_core/paperless/client.py` next to the existing aux tags. Live install bootstrap ran cleanly — tag id 96 in Paperless.

2. **`services/auto-tagger/src/auto_tagger/dedup.py`** — pure detector module with a `DocFields` dataclass and a `find_duplicates(new_doc, candidates) -> list[int]` function. Short-circuits on missing `ai_correspondent` or `ai_issue_date` (without those anchors the false-positive rate climbs); excludes self-id; OR semantics across the monetary + reference-number signals; ISO-currency-prefix-aware amount parsing; casefold-normalised correspondent + reference-number comparison (handles German ß → ss via `casefold()`).

3. **`packages/aktenraum-core/src/aktenraum_core/paperless/client.py`** — `get_documents_with_tag` extended with an `extra_params: dict | None` parameter so the propagator can filter to one correspondent at the Paperless query layer (avoiding O(N) Python-side filtering).

4. **`services/auto-tagger/src/auto_tagger/propagator.py`** — `process_approved_document` now runs detection BEFORE the lifecycle PATCH so the new doc's `ai-duplicate` lands in the same write (one PATCH, not two). Per-matched-id `add_tag_to_document` is wrapped in try/except — both detection-lookup failures and per-id tag-PATCH failures are logged at warning and never break propagation.

5. **`services/aktenraum-api/src/aktenraum_api/library/service.py`** — `_BADGE_TAGS` includes `ai-duplicate` so flagged rows render the purple pill. Intentional asymmetry with `ai-auto-approved`: NOT added to `_INTERNAL_TAGS` because the user explicitly wants to filter Library by `?tags=ai-duplicate` to find candidates.

## Follow-up commit — type-specific fields save regression (May 17, afternoon)

**Focus:** Fix two bugs introduced or exposed by the "type-specific fields react immediately to dropdown change" fix shipped earlier today.

**Repo state:** branch `main`; 588 tests pass; ruff + SPA build clean.

### Bugs fixed

**Bug 1 — 422 on type-specific save when dropdown type differs from Paperless type**

`PATCH /documents/{id}/type-fields` always inferred the document type from the Paperless `ai_document_type` custom field. After the dropdown fix, the frontend shows fields for the new type immediately but Paperless still has the old type — so saving Zeugnis fields against a document Paperless still considers "Behördenbrief" (or "Sonstiges") raised a 422.

Fix:
- `type_fields/schemas.py`: added `document_type: str | None = None` to `TypeFieldsPatch`
- `type_fields/router.py`: prefers `body.document_type` over the Paperless infer
- `type_fields/service.py`: `upsert` accepts `doc_type_str` as a parameter (no double-infer); when the type changes, old-type fields are purged from the DB row instead of accumulating silently
- `apps/web/src/lib/typeFields.ts`: `patchTypeFields` and `usePatchTypeFields` now accept and forward `documentType` in the request body
- `apps/web/src/components/TypeSpecificFieldsSection.tsx`: `handleSave` passes `effectiveDocumentType` to the mutation

**Bug 2 — "nothing saved" on reload after saving type-specific fields without saving the main form**

The section used `form.ai_document_type || detail.data.ai_document_type` (the Paperless value) to decide which type's fields to display. If the user saved Zeugnis type-specific fields but never saved the main form, Paperless still reported the old type, so on reload the section displayed the wrong (empty) fields — even though the DB had correct data.

Fix:
- `TypeSpecificFieldsSection` now accepts a separate `fallbackDocumentType` prop and computes `effectiveDocumentType = documentType (explicit override) || dbDocumentType (our DB) || fallbackDocumentType (Paperless)`
- Parents pass `documentType={form.ai_document_type !== detail.data.ai_document_type ? form.ai_document_type : null}` (null = no override) and `fallbackDocumentType={detail.data.ai_document_type}`
- Removed `documentType` from the `useEffect` dependency array (the form reset on type change was unnecessary — `handleSave` already filters payload to current-type fields only)
