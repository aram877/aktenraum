## Session addendum — testing-phase remote access (OpenSpec change `tailscale-remote-access`)

### Context

The maintainer wanted to validate the product from mobile + a second laptop without exposing aktenraum to the public internet. Tauri Phase 0 is premature for the testing phase (packaging effort against an unvalidated product). Brainstormed three topologies (Tailscale private mesh, Cloudflare Tunnel + Access, VPS with Anthropic backend) and picked Tailscale: zero public exposure, no third-party traffic visibility, free for personal use, works on iOS / Android / Windows / macOS / Linux, and — critically — zero code changes are required because the existing CSRF middleware, nginx `server_name _;`, and `COOKIE_SECURE` default already accommodate it.

### What shipped

#### OpenSpec change `tailscale-remote-access`

`openspec/changes/tailscale-remote-access/proposal.md`  
`openspec/changes/tailscale-remote-access/design.md`  
`openspec/changes/tailscale-remote-access/specs/remote-access/spec.md`

Full proposal + design + spec + tasks. The change introduces a new `remote-access` capability and modifies no existing capabilities (no code changes to aktenraum-api / nginx / auto-tagger).

#### ADR-005 + ADR-002 cross-link

`docs/adr/005-test-phase-access-via-tailscale.md` (new)  
`docs/adr/002-distribution-desktop-app.md` (status line only)

ADR-005 records the testing-phase access topology as Tailscale-on-host and explicitly defers ADR-002's Tauri Phase 0 work behind four falsifiable milestones: 30 days of sustained mobile use, ≥200 propagated docs across ≥10 document types, RAG recall@5 ≥ 0.7 + MRR ≥ 0.5 on a ≥20-question golden set, and at least one external buyer signal. ADR-002 stays Accepted with its status line now reading "Accepted (Phase 0 deferred — see ADR-005)".

#### Runbook

`docs/runbooks/tailscale-remote-access.md` (new, ~250 lines)

Phase A → G end-to-end setup walkthrough with Windows / macOS / Linux variants, target wall-clock 15 min for first-time setup. Eight-step main path: install Tailscale on host → `tailscale up` → `tailscale status` → `tailscale serve --bg --https=443` → `tailscale serve status` → install Tailscale on client → visit MagicDNS URL → log in. Two branches (Linux headless server, household sharing), five failure-mode entries (cert warning / MagicDNS off, DNS not resolving, ACL blocking, cookie-not-sent footgun, 404 from `tailscale serve`), and a documented rollback (`tailscale serve --remove`).

#### `.env.example` documentation

`docker/aktenraum-api.env.example`

Replaced the existing terse `COOKIE_SECURE` comment with an explicit three-scenarios block: (1) Tailscale HTTPS = `true` default, (2) localhost dev = `false` override, (3) explicit "do not flip false to fix LAN-IP plain-HTTP" footgun warning.

#### Taskfile shortcuts

Added `task tailscale:serve` and `task tailscale:status` under a new "Tailscale remote access" section. The `serve` task reads `${AKTENRAUM_WEB_PORT:-8080}` so the port override path stays consistent.
