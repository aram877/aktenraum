## Session addendum — self-service password change (OpenSpec change `auth-password-change`)

### Context

The bootstrap-seed flow ran once on first install; after that the only way to rotate the password was to UPDATE `users.password_hash` via psql or DELETE + recreate the container. The user (rightly) called this out as a footgun and asked for option C from a brainstorm where I'd offered three options (psql UPDATE / bootstrap-reseed / proper endpoint + UI). This change ships option C.

### What shipped

#### Backend endpoint `POST /api/auth/change-password`

`services/aktenraum-api/src/aktenraum_api/auth/schemas.py`  
`services/aktenraum-api/src/aktenraum_api/auth/router.py`

Auth-gated (cookie required). Body: `{current_password, new_password}` with Pydantic min/max length bounds (current 1..128, new 8..128). Verifies the current password against `user.password_hash`; rejects `new == current` with 400. On success: bcrypt-hashes the new password, commits, clears the session cookie so the device is forced into re-auth. Returns 204.

The current-password check is the load-bearing defence against the stolen-cookie scenario: an attacker with the cookie alone cannot lock the legitimate owner out by silently rotating the password. CSRF middleware applies automatically (it's a POST inside the existing perimeter).

#### Tests (5 new)

`services/aktenraum-api/tests/test_auth_flow.py`

- `test_change_password_happy_path`: change → cookie cleared → /me 401 → old password fails → new password works.
- `test_change_password_wrong_current`: 401, cookie intact, password unchanged.
- `test_change_password_new_equals_current`: 400, cookie intact, password unchanged.
- `test_change_password_unauthenticated`: 401, no DB write.
- `test_change_password_too_short`: 422 (Pydantic), cookie intact.

#### SPA

`apps/web/src/lib/api.ts` — `changePassword(current, new)` POST helper.  
`apps/web/src/lib/auth.ts` — `useChangePassword()` mutation; on success invalidates `ME_KEY` so the SPA reflects the logged-out state immediately.  
`apps/web/src/routes/Settings.tsx` — new `KontoSection` rendered above the existing model pickers, separated by a `border-t border-hairline`. Three password fields (current / new / confirm) with `autoComplete="current-password" | "new-password"` set per W3C convention so the browser still offers to save the new password. Client-side validation blocks submit when confirm mismatches or new is too short. On success: clears fields, renders a green "Passwort geändert — du wirst zum Login geleitet." banner, then `setTimeout(() => navigate({to: "/login"}), 1500)`. Error mapping: 401 → "Aktuelles Passwort ist nicht korrekt.", 400 → "Das neue Passwort muss sich vom aktuellen unterscheiden.", 422 → "Bitte fülle alle Felder korrekt aus (min. 8 Zeichen für das neue Passwort).", other → generic fallback.

### Known v1 limitations (documented in design.md, NOT bugs)

- **JWTs on OTHER devices stay valid until their `exp`** (default 8h). The cookie-clear pattern handles the current device; instant kill-all would need either `JWT_SECRET` rotation (sledgehammer; logs out all users) or a per-user `password_version` JWT claim (clean but requires a User schema migration). Defer to v2 if it becomes a real complaint.
- **No forgot-password flow.** No SMTP. The original psql UPDATE recovery path stays as the lost-password story.

## Session addendum — mobile document scan, Phase 1 (OpenSpec change `mobile-document-scan`)

### Context

The mobile-responsive layout shipped last week and the Tailscale topology makes the SPA reachable from a phone, but the only ingest path from mobile was the standard `<input type="file">` picker — which means "switch to camera app → take photo → switch back → pick from camera roll → upload" for every page. Multi-page documents (contracts, salary slips) compounded the friction because each capture landed as a separate document. The user wanted an in-app scanner mirroring Genius Scan / Apple Notes: open scanner → frame → snap → next page → review → done → one PDF in the inbox.

The scope brainstorm landed on a deliberate two-phase build: Phase 1 keeps the OS camera UI (via `<input capture>`) and adds client-side PDF composition, reorder/rotate/crop/delete, and the existing upload pipeline; Phase 2 lifts UX to "competitive with Adobe Scan" by lazy-loading `jscanify` + OpenCV.js for live edge-detection and perspective correction. Phase 1 ships first so we learn from real use whether Phase 2 is worth the ~8 MB chunk.

### What shipped (Phase 1)

#### OpenSpec change `mobile-document-scan`

`openspec/changes/mobile-document-scan/proposal.md`  
`openspec/changes/mobile-document-scan/design.md`  
`openspec/changes/mobile-document-scan/specs/aktenraum-web/spec.md`  
`openspec/changes/mobile-document-scan/tasks.md`

Full proposal + design + spec + tasks. Phase 2 lives in the same change as task group 6 — separated cleanly so the Phase-2 PR is purely additive (feature flag + dynamic import wiring), and Phase 1 doesn't get held up by edge-detection scope.

#### New SPA route `/scan`

`apps/web/src/lib/scan-types.ts`  
`apps/web/src/lib/scan-reducer.ts`  
`apps/web/src/lib/scan-pdf.ts`  
`apps/web/src/routes/Scan.tsx`  
`apps/web/src/router.tsx`  
`apps/web/src/components/Nav.tsx`  
`apps/web/src/components/Icons.tsx`

`scan-reducer.ts` is a pure `useReducer` covering six actions (`add` / `remove` / `rotate` / `reorder` / `crop` / `replace`), with discriminated-union types in `scan-types.ts` (`ScanPage`, `ScanAction`, `ScanState`, `MAX_PAGES=30`). Reducer is reference-stable on no-op (e.g. `remove` of a non-existent id returns the same state ref) and clamps reorder indices to valid range.

`scan-pdf.ts` is the pure PDF composer: `pagesToPdf(pages, { jpegQuality?, maxSidePx? })` decodes each blob → draws onto an off-screen canvas with the page's rotation + crop applied → downscales if longest side > 3200 px → encodes JPEG at quality 0.85 (defaults) → embeds via `pdfDoc.embedJpg` → adds an A4-portrait page with the embedded image letterboxed to preserve aspect ratio. Output: `Blob('application/pdf')`. Also exports `formatLocalIso(date)` for filename defaults.

`Scan.tsx` is the route component. Single hidden `<input type="file" accept="image/*" capture="environment">` triggered by a "Seite aufnehmen" button — on iOS Safari + Android Chrome this invokes the OS camera; on desktop it falls back to the file picker (same code path, no separate desktop branch). Thumbnail grid (2 cols mobile / 3 cols sm+) with per-tile controls: ↑↓ reorder buttons (chose arrow buttons over `dnd-kit` for the MVP — simpler, mobile-friendly, no new dep), rotate (90° CW step), crop (opens full-screen `react-image-crop` modal), delete. Filename input below the grid defaults to `scan-YYYY-MM-DD-HHmmss`. The "Hochladen" button composes the PDF via `pagesToPdf`, wraps in `new File([blob], filename + '.pdf', { type: 'application/pdf' })`, and calls the existing `uploadDocument({file, onProgress})` from `lib/documents.ts` — same endpoint, same multipart shape, same response.

Post-upload polling reuses the exact pattern from `Upload.tsx`: poll `/task/{uuid}` every 1500ms until SUCCESS, then poll `/{doc_id}/status` every 3000ms until a lifecycle tag appears (with 120s ceiling). Phases: `idle → composing → uploading → consuming → ai → (inbox | library | error)`. Terminal state renders a summary card with "Weiteres Dokument scannen" / "Zur Prüfung →" / "Zur Bibliothek →" / Library link to the new doc.

Nav: new "Scannen" link next to "Hochladen" in both desktop nav and mobile drawer. New `NavKey` value `"scan"`.

Crop modal: `react-image-crop` (MIT, hooks-native, ~20 KB — swapped from the originally-spec'd `cropperjs` because react-image-crop is React-first and the cropperjs path would have required a `react-cropper` wrapper for the same outcome at ~50 KB combined). Full-screen modal; "Übernehmen" dispatches `{type: 'crop', id, crop: {x, y, w, h}}`; "Zurücksetzen" clears any prior crop.

#### Deliberate scope choices

- **Reorder via ↑↓ buttons, not drag-and-drop.** `dnd-kit` is the right pick for drag on mobile but adds a dependency for what's essentially a 5-line UI affordance. Buttons are clearer and work the same on touch + mouse. Drag-to-reorder is a phase-2 polish if it becomes a request.
- **No unit tests (tasks 1.4, 2.3 deferred).** The rest of `apps/web` has zero unit tests — the SPA is verified via `pnpm build` + `pnpm lint` + manual smoke. Adding `vitest` + `jsdom` just for this feature creates inconsistent precedent. The reducer is small enough to verify by reading; `pagesToPdf` is verified by the manual smoke (open `/scan`, capture, upload, check the PDF in the Library preview).
- **No draft persistence across navigations.** Backing out of `/scan` mid-capture loses the page list. Acceptable for v1 per the design doc's non-goals.
- **Filename default ISO timestamp, no edit-after-upload.** User edits before upload only. Renaming a doc post-upload is already possible via Paperless admin / Library detail page.
