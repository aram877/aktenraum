## Bugs caught during live testing

Brought up the local stack and ran the script end-to-end against Paperless 2.20.15. Two enum values failed Paperless's API validation â€” both my initial values were wrong, fixed in-flight:

1. **`attachment_type=3` rejected** (`"3" is not a valid choice`). `OPTIONS /api/mail_rules/` revealed Paperless 2.20 only supports 1 (attachments only) and 2 (all incl. inline) â€” no 3. Changed default to 2.
2. **`action=6` (MARK_AS_PROCESSED) doesn't exist** in this version. The Paperless wiki mentions an internal "mark as processed" flag, but `OPTIONS` says action choices are 1=DELETE, 2=MOVE, 3=MARK_READ, 4=FLAG, 5=TAG. Removed `MARK_AS_PROCESSED` from the script's enum mapping AND from `.env.example` AND the runbook. MARK_READ has the same effective behaviour because the rule filter skips read mails on subsequent polls.

Lesson recorded for future Paperless integrations: **hit `OPTIONS /api/<endpoint>/` first for the authoritative enum list**. Cheaper than guessing from the SDK and surviving across version bumps. Pattern is reusable for any future "provision X via the API" script.

- **Multi-mailbox support**: the current design assumes one mailbox. A user with `bills@`, `medical@`, and `personal@` would need to re-run the script with different `AKTENRAUM_MAIL_NAME` values; the second run would create a *second* account+rule pair (because `lookup_id_by_name` is name-keyed). Captured here so it doesn't drift.
