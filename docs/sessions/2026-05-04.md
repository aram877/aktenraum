## What we shipped this session, by feature

### 1. Tags as a first-class filter dimension (`f91beaa`)

The user uploaded a CV that the auto-tagger misclassified as `Arbeitszeugnis` while correctly emitting `Lebenslauf` as a tag. Promoted tags from passive metadata to a filter so the user can find docs the doc_type taxonomy gets wrong.

- Backend: `GET /api/library/tags` returns top tags by frequency among non-pending docs (sample 500, min count 2). Lifecycle / aux tags excluded from the user-facing vocabulary.
- `?tags=foo&tags=bar` on `/api/library/` filters with AND semantics via Paperless `tags__id__all`. Unknown tag names short-circuit to zero results.
- `SearchFilter.tags: list[str]` flows through `/api/ai/find` and `/api/ai/answer`. AI prompts now include the live tag vocabulary (cap 200), with a German hint that tags are reliable when doc_type is ambiguous, plus a "Mein Lebenslauf" few-shot exemplar.
- `_broaden_for_answer` treats tags as structural (drops free text from the filter when only tags are set).
- Frontend: Library sidebar tag-cloud facet with counts; row tag chips; Find/Ask filter chips render tag-removals as separate green chips.
- 14 new tests.

### 2. Answer-pipeline robustness (`5e46f62`)

User hit two failure modes on `/ask`: small local LLM emitted `{"answer_<|channel|>{": "…"}` (control-token leak in JSON keys), then on retry emitted the literal `"answer_de"` as the answer text. Both surfaced as either a 422 or as the field name displayed verbatim. Three layered defences:

1. `OllamaBackend.complete` retries validation once after best-effort key recovery. When a canonical schema field is missing AND exactly one sibling key shares its underscore-prefix, rename it (`answer_<|channel|>{` → `answer_de`). Sentinel test pins this.
2. `/api/ai/answer` turns `ValidationError` into a 200 with retrieved candidates as citations + a German soft-fail message ("Ich konnte die Antwort nicht zuverlässig formulieren…") instead of bouncing the whole request.
3. Degenerate-answer guard catches schema-echo (`answer_de`, `answer`, `antwort`, `string`); routes through the same soft-fail. As a follow-up, when prose is real but `cited_ids` is empty, citations are backfilled from the retrieval set so the user always has a source to verify.

5 unit tests + 3 router tests.

### 3. Loading state on `/ask` (`4435722`)

The 30s wait with only "…" on the button was confusing. Added a skeleton card while the mutation is pending: spinner + "Denke nach…" + an explanation that two LLM calls run + three pulsing skeleton lines where the answer will land. Replaced by the real answer when the mutation resolves. ARIA-busy + aria-live for screen readers.

### 4. Nginx DNS resolver fix (`87c3965`)

After rebuilding `aktenraum-api`, login broke with 502. nginx had cached the old upstream IP at startup; rebuilds give the api container a new IP. Fix: Docker's embedded DNS resolver (`resolver 127.0.0.11 valid=10s`) plus a variable in `proxy_pass` so nginx re-resolves per request. Required appending `$request_uri` because variable-form `proxy_pass` no longer auto-forwards the URI. Eliminates the rebuild-breaks-login footgun.

### 5. Few-shot exemplars enabled by default (`ea3c4eb`)

`FEW_SHOT_EXAMPLES` bumped from 0 to 3 in `auto-tagger.env.example`. Anchors every fresh extraction on three of the user's most-recently propagated docs. Cheapest fix for the doc_type misclassifications that drove the tags-as-filter feature in the first place. +1.5–2k tokens / +30% latency per extraction; negligible at personal-DMS scale.

### 6. SSE streaming for `/ask` (`c899b76`)

Replaces the silent ~30s wait + single JSON drop with token-by-token streaming. Perceived latency drops from "blank, then 30s, then everything" to "first chunk in ~1s, prose builds visibly".

- aktenraum-core: new `LLMBackend.stream_text(messages) -> AsyncIterator[str]` Protocol method. Implemented in `OllamaBackend` (chat with `stream=True`) and `AnthropicBackend` (`messages.stream`, splits role:system into the top-level kwarg).
- aktenraum-api: new `POST /api/ai/answer/stream` returning `text/event-stream`. Events: `meta` (filter + explanation + total) → 0..N `chunk` (text deltas) → `final` (full prose + resolved citations + total). On filter/retrieval error: `error`. Streaming-specific prompt asks for inline `[Quelle: <id>]` markers; we regex them out post-hoc and intersect with the retrieved set, falling back to retrieval candidates if nothing was cited. All robustness layers (degenerate detection, soft-fail) carry over.
- apps/web: new `streamAsk(question, handlers)` using fetch + ReadableStream (EventSource is GET-only; question lives in the POST body). Returns an AbortController so callers can cancel in-flight streams. Ask.tsx renders incoming text with a blinking caret; on `final`, replaces accumulated text with the canonical value (so soft-fail rewrites land cleanly). Inline `[Quelle: N]` markers stripped client-side.
- nginx: dedicated `location = /api/ai/answer/stream` block with `proxy_buffering off`, `proxy_cache off`, `chunked_transfer_encoding on`. Other JSON endpoints keep their default buffering.
- 5 SSE tests via a scripted streaming backend.

### 9. RAG Phase 1 design doc (`bdc95b9`)

Architecture for production-grade local retrieval. Decisions locked in writing with rationale against alternatives:
- **Qdrant** (new container, dedicated) over pgvector. Hybrid + payload filtering in a single query; production-grade past 100k chunks.
- **bge-m3** via Ollama. The only widely-deployed open model that emits dense + sparse from the same forward pass. Multilingual, German-strong, 1024-dim.
- **bge-reranker-v2-m3** cross-encoder. Largest quality lever after basic retrieval. Run via `sentence-transformers` until Ollama supports rerankers.
- **Paragraph-aware chunking**, ~500 tokens with ~50-token overlap. Sentence fallback for oversize paragraphs.
- **Hybrid query time**: dense + sparse via Qdrant → RRF fusion → reranker → top-5 chunks → answer LLM. Structural filters apply at the Qdrant payload layer.
- **Indexing pipeline**: a fifth concurrent task in auto-tagger triggered when a doc reaches `ai-propagated`. Failures tag `ai-index-error`. Reprocess deletes existing chunks first (idempotent).
- **Eval harness**: YAML golden questions + recall@K and MRR runner. CI blocks regressions.

12-step sub-phasing in the doc; ~5 working days estimated total.

### 10. RAG 1.1 — chunker (`2e3e15b`)

Pure-function paragraph-aware splitter in `aktenraum-core/src/aktenraum_core/rag/chunker.py`. Pack paragraphs greedily into ~500-token chunks; if a paragraph alone exceeds target, fall back to sentence-level splits on `[.!?]+\s+[A-ZÄÖÜ0-9]` (uppercase lookahead avoids false splits on `z. B.` and similar German abbreviations). 50-token overlap between chunks; first chunk's `char_start` reaches into the previous chunk's tail so SPA highlights stay accurate. Token counting is whitespace-split (cheap, ~25% loose vs real BPE — acceptable since bge-m3's 8192-token context dwarfs our target).

Edge case pinned by tests: `MIN_CHUNK_TOKENS = 5` applies only to chunks AFTER the first. The first chunk of a very short document (a one-line CV) lands regardless — the threshold is for filtering overlap-only stub follow-ons, not whole short docs.

23 unit tests.

### 11. RAG 1.2 — embedder (`91c5259`)

`OllamaEmbedder` in `aktenraum-core/src/aktenraum_core/rag/embedder.py`. Wraps `ollama.AsyncClient.embed` for batched dense embeddings. Targets `bge-m3` by default (1024-dim multilingual).

Sparse output (the bge-m3 SPLADE-like vectors that catch exact-string queries — invoice numbers, German company names) is **not yet shipped**: Ollama doesn't expose them for any model. v1 is dense-only; the `Embedder` Protocol is shaped so the eventual sparse path can plug in by adding `embed_sparse` to the interface without touching any existing call site.

8 unit tests via a stub `_FakeOllamaClient`.

### 12. RAG 1.3 — Qdrant vector store wrapper (`3e9de11`)

`QdrantVectorStore` in `aktenraum-core/src/aktenraum_core/rag/vector_store.py`. Thin async façade over `qdrant-client.AsyncQdrantClient`:

- `ensure_collection` — create if missing, ensure payload indexes (idempotent; "already exists" errors on indexes are swallowed).
- `upsert_chunks` — `(doc_id, chunk_index, vector, payload)`. Point IDs are deterministic UUID5s of `(doc_id, chunk_index)` so re-indexing replaces rather than duplicates. Payload denormalises doc-level metadata (doc_type, correspondent, tags, created_date) onto every chunk so structural filters apply at the Qdrant layer instead of post-fetch.
- `delete_by_doc_id` — clean up before re-indexing.
- `search` — uses `query_points` (Qdrant 1.10+); `SearchFilter` composes `MUST` (AND across fields) + `MatchAny` (OR within a field).
- `health_check` — non-throwing, returns False on error.

Sentinel test pins the namespace UUID — regenerating it would orphan every chunk in a deployed collection.

Adds `qdrant-client>=1.12.0` to `aktenraum-core` deps. 21 unit tests via a stub `_FakeQdrantClient`.

### 13. RAG 1.4 — Qdrant container in compose (`e692196`)

New `qdrant` service on `qdrant/qdrant:v1.17.1`. Persistent volume at `~/aktenraum/qdrant` (will move to `${AKTENRAUM_DATA_DIR}/qdrant` once Phase 0.2 lands). REST 6333 + gRPC 6334 exposed inside the network only. TCP-level healthcheck via bash's `/dev/tcp` (the qdrant image has bash but no wget/curl; `CMD ["bash", "-c", ...]` because `CMD-SHELL` routes through dash which lacks `/dev/tcp`).

`auto-tagger` and `aktenraum-api` gain `qdrant: condition: service_healthy` in `depends_on`. `QDRANT_URL` defaults to `http://qdrant:6333` in compose so an empty value in env files Just Works.

**Live verified end-to-end** (smoke from inside `aktenraum-api` against the running qdrant): `ensure_collection` creates the collection + indexes; `upsert_chunks` writes 1 point; `search` finds it with score=1.0; `health_check` returns True; `delete_by_doc_id` clears it. The RAG primitives are operational on the dev stack.

- bge-m3 sparse output via Ollama not yet available — v1 ships dense-only with a Protocol seam for future sparse.
