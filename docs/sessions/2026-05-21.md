## Session addendum — testing-phase remote access (OpenSpec change `tailscale-remote-access`)

### Context

The maintainer wanted to validate the product from mobile + a second laptop without exposing aktenraum to the public internet. Tauri Phase 0 is premature for the testing phase (packaging effort against an unvalidated product). Brainstormed three topologies (Tailscale private mesh, Cloudflare Tunnel + Access, VPS with Anthropic backend) and picked Tailscale: zero public exposure, no third-party traffic visibility, free for personal use, works on iOS / Android / Windows / macOS / Linux, and — critically — zero code changes are required because the existing CSRF middleware, nginx `server_name _;`, and `COOKIE_SECURE` default already accommodate it.

## Session addendum — self-service password change (OpenSpec change `auth-password-change`)

### Context

The bootstrap-seed flow ran once on first install; after that the only way to rotate the password was to UPDATE `users.password_hash` via psql or DELETE + recreate the container. The user (rightly) called this out as a footgun and asked for option C from a brainstorm where I'd offered three options (psql UPDATE / bootstrap-reseed / proper endpoint + UI). This change ships option C.

### What shipped

#### Backend endpoint `POST /api/auth/change-password`

`services/aktenraum-api/src/aktenraum_api/auth/schemas.py`  
`services/aktenraum-api/src/aktenraum_api/auth/router.py`

Auth-gated (cookie required). Body: `{current_password, new_password}` with Pydantic min/max length bounds (current 1..128, new 8..128). Verifies the current password against `user.password_hash`; rejects `new == current` with 400. On success: bcrypt-hashes the new password, commits, clears the session cookie so the device is forced into re-auth. Returns 204.

The current-password check is the load-bearing defence against the stolen-cookie scenario: an attacker with the cookie alone cannot lock the legitimate owner out by silently rotating the password. CSRF middleware applies automatically (it's a POST inside the existing perimeter).

#### Tests (5 new)

`services/aktenraum-api/tests/test_auth_flow.py`

- `test_change_password_happy_path`: change → cookie cleared → /me 401 → old password fails → new password works.
- `test_change_password_wrong_current`: 401, cookie intact, password unchanged.
- `test_change_password_new_equals_current`: 400, cookie intact, password unchanged.
- `test_change_password_unauthenticated`: 401, no DB write.
- `test_change_password_too_short`: 422 (Pydantic), cookie intact.

#### SPA

`apps/web/src/lib/api.ts` — `changePassword(current, new)` POST helper.  
`apps/web/src/lib/auth.ts` — `useChangePassword()` mutation; on success invalidates `ME_KEY` so the SPA reflects the logged-out state immediately.  
`apps/web/src/routes/Settings.tsx` — new `KontoSection` rendered above the existing model pickers, separated by a `border-t border-hairline`. Three password fields (current / new / confirm) with `autoComplete="current-password" | "new-password"` set per W3C convention so the browser still offers to save the new password. Client-side validation blocks submit when confirm mismatches or new is too short. On success: clears fields, renders a green "Passwort geändert — du wirst zum Login geleitet." banner, then `setTimeout(() => navigate({to: "/login"}), 1500)`. Error mapping: 401 → "Aktuelles Passwort ist nicht korrekt.", 400 → "Das neue Passwort muss sich vom aktuellen unterscheiden.", 422 → "Bitte fülle alle Felder korrekt aus (min. 8 Zeichen für das neue Passwort).", other → generic fallback.

### Known v1 limitations (documented in design.md, NOT bugs)

- **JWTs on OTHER devices stay valid until their `exp`** (default 8h). The cookie-clear pattern handles the current device; instant kill-all would need either `JWT_SECRET` rotation (sledgehammer; logs out all users) or a per-user `password_version` JWT claim (clean but requires a User schema migration). Defer to v2 if it becomes a real complaint.
- **No forgot-password flow.** No SMTP. The original psql UPDATE recovery path stays as the lost-password story.
