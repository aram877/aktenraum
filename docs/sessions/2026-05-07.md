## What we shipped this session, by sub-phase

### RAG 1.5 — auto-tagger indexer task (`b96c0dc`)

A fifth concurrent asyncio task in the auto-tagger that fans out from `ai-propagated` → chunk → embed → upsert to Qdrant. Composes the chunker (1.1), embedder (1.2), and Qdrant wrapper (1.3) into one coroutine.

- New `services/auto-tagger/src/auto_tagger/indexer.py` with `index_document(doc_id, deps)`.
- Propagator hooks: on success, enqueues `doc_id` on `indexing_queue` (non-blocking `put_nowait`; `QueueFull` logs and drops).
- Idempotent: every run does `delete_by_doc_id` first so a re-index never duplicates.
- Failure tags `ai-index-error` (auxiliary, NOT in `LIFECYCLE_TAGS`); success self-heals — clears `ai-index-error` if present.
- 200-chunk-per-doc cap as a runaway-OCR guard.
- Opt-in via `QDRANT_URL`: empty → no vector store, no worker, no enqueue. Existing extraction + propagation paths untouched.
- 13 unit tests + live verification: doc 25 reprocessed → propagated → indexed → `chunks_written=1`, `content_chars=2895`. Qdrant search for `doc_ids=[25]` returns the chunk verbatim.

### RAG 1.6 — backfill script (`8308978`)

Iterates every `ai-propagated` doc not yet in Qdrant, runs the same `index_document` path. Resumable + idempotent. Without this, only newly-propagated docs would be searchable; the existing corpus needed a one-shot.

- `services/auto-tagger/src/auto_tagger/backfill.py`:
  - `iterate_propagated_doc_ids` — paginates `/api/documents/?tags__id__all=<ai-propagated>` (the existing `get_documents_with_tag` returns only the first page).
  - `backfill_index(deps, *, force=False)` — two-pass (count, then work) so the `started` event carries a total for progress UIs.
  - Per-doc verdicts: `doc_skipped` (existing chunks > 0), `doc_indexed` (succeeded), `doc_failed` (zero chunks landed OR per-doc exception).
- CLI: `python -m auto_tagger.backfill [--force]`. JSON-line events to stdout, flushed per record so the desktop shell sees progress incrementally.
- New `QdrantVectorStore.count_chunks_for_doc(doc_id)` using Qdrant's count API (cheap — no vectors returned).
- `scripts/backfill-rag-index.sh` — thin shell wrapper around `docker compose exec`.
- 12 unit tests + 2 for the count helper.
- Live verified: 18 docs indexed in ~7 seconds, 1 skipped (doc 25 from 1.5's smoke), 0 failed. Re-run was a clean no-op (indexed=0, skipped=19, failed=0).

### RAG 1.7 — reranker module (`b861162`)

Cross-encoder reranking is the single largest quality lever in RAG. Dense top-K returns plausibly-relevant; the reranker re-scores each (query, candidate) pair through a transformer that sees BOTH sides jointly.

- `aktenraum_core.rag.reranker`:
  - `Reranker` Protocol (mirrors the `Embedder` shape).
  - `LocalReranker` wraps `sentence_transformers.CrossEncoder`. Default model: `BAAI/bge-reranker-v2-m3` (~600 MB, multilingual, ~50 ms/pair on CPU).
  - **Lazy load**: model constructed on the first `rerank()` call. Tests build a LocalReranker without paying a 5s model load; the 600 MB HF download happens on first use, surfaced via the `reranker_loading_model` log so the desktop shell can render a "downloading models" UI.
  - Soft import inside `_ensure_loaded` — packages that don't construct a LocalReranker don't pay the install weight.
  - Sync `predict` → `asyncio.to_thread` so FastAPI's event loop stays unblocked.
- Output: `RerankCandidate(id, text)` → `RerankResult(id, score)`, sorted descending. `numpy.float32` scores coerced to plain `float` for JSON serialisation.
- `sentence-transformers>=3.3.0` added to `aktenraum-api` only (not aktenraum-core) so the auto-tagger image stays slim.
- 14 unit tests via a stub CrossEncoder.

### RAG 1.8 — hybrid retrieval coordinator + lifespan wiring (`fbaca22`)

The composition layer. `aktenraum_api.ai.retrieval.retrieve_chunks_for_question` chains the four primitives into one path: embed → Qdrant search w/ payload filter → rerank → top-N RetrievedChunk[].

- New `RetrievalDeps` bag and `RetrievedChunk` frozen dataclass.
- `Settings` gains `qdrant_url`, `embedding_model`, `reranker_model`, `rag_retrieval_top_k` (50), `rag_rerank_top_k` (5).
- Lifespan startup constructs the deps ONCE per process when `QDRANT_URL` is set — the embedder + Qdrant client + LocalReranker pin to `app.state` so we don't re-pay setup costs per request. Lifespan shutdown `aclose`s the Qdrant client.
- `get_retrieval_deps` FastAPI dependency returns the deps or None — endpoints branch on None to fall back when RAG is disabled or Qdrant was unreachable at boot.
- Resilience: empty/whitespace question short-circuits with no upstream calls. Each per-stage failure (embedder / qdrant / reranker) degrades gracefully — reranker error falls through to dense-only ordering rather than failing the whole request.
- LLM `SearchFilter` projection: `document_type` / `correspondent` / `tags` map to the indexed Qdrant payload fields. Date ranges and amount bounds aren't indexed on chunks; those apply post-fetch on the existing structural answer path.
- 13 unit tests covering empty / no-hits / per-stage error / structural-filter projection.

### RAG 1.9 — wire into `/api/ai/answer/stream` (`a2894bd`)

The actual user-visible change: the streaming answer endpoint now feeds reranked chunks (with `[Quelle: <doc_id>]` markers) into the answer LLM's prompt alongside the existing AI metadata.

- `_stream_answer_events` gains an optional `retrieval_deps` parameter.
- New step 3a: `retrieve_chunks_for_question` runs hybrid retrieval with the LLM-emitted `SearchFilter` applied as Qdrant payload filter.
- Step 3b: `_ranked_unique_doc_ids` dedupes chunks by doc_id preserving rank order; `_reorder_or_fetch` matches the order against the structural results, fetching minimal `DocumentSummary` for any RAG-only doc so it can still surface as a citation card.
- Step 3c: `_group_chunks_by_doc` buckets chunks per doc (cap 3) and feeds them through `build_streaming_answer_messages` as `chunks_by_doc=...`.
- Citation pool extended: `final` event resolves cited ids against the union of structural + RAG-promoted docs.
- **Crucial fix**: the previous `_render_candidate` clipped each chunk to 2000 chars. The first live test failed because the Kopfstand mention sat at chars ~2400 of a 2878-char chunk and was being clipped off. Lifted the per-chunk cap entirely — the chunker already bounds individual chunks to ~500 tokens (~3 KB) by design, so re-clipping in the prompt was overcautious.
- `rag_retrieve_complete` info log (counts of candidates, reranked, returned, top doc_ids) for ops visibility.

**Live verified**: *"Wie lange habe ich bei Kopfstand gearbeitet?"* → *"Du hast bei Kopfstand vom 05.2020 bis 12.2020 als Webentwickler gearbeitet. [Quelle: 25]"* with correct citation. Pre-RAG: hit the soft-fail path. The capability gap RAG Phase 1 was closing is now closed end-to-end.

### RAG 1.10 — eval harness (`1328fc3`, `e267c0b`)

The credibility piece. Without measurements, "best-in-class retrieval" is unsubstantiated marketing. With it, every prompt / model / chunker change is evaluable against a stable yardstick and regressions are visible before they ship.

Two pieces:

- `aktenraum_api/eval/metrics.py` — pure-function metrics: `score_case` (per-case `recall@K` + `reciprocal_rank` with multi-expected first-match-wins) and `aggregate` (rolls into `EvalReport` with recall@K and MRR aggregates). Empty input returns zeros rather than dividing by zero.
- `aktenraum_api/eval/runner.py` — live runner: `load_cases` parses YAML with strict schema validation; `evaluate` runs every case through `retrieve_chunks_for_question` against the live deps; `render_text` for terminal eyeballing + `render_json` for diffing across runs / CI threshold gates.

Wrapping:
- `scripts/run-rag-eval.sh` — `docker compose exec` thin wrapper.
- `evals/golden-questions.yaml` — three starter cases tied to dev-corpus ids (committed file documents the schema; buyers re-pin against their own ids).
- `docker-compose.yml` — bind-mounts `../evals:/app/evals:ro` so editing YAML doesn't require a rebuild.

29 unit tests covering metrics math (boundary cases, multi-expected, empty input, frozen dataclasses) and runner orchestration (YAML schema enforcement, evaluate aggregation, render round-trip).

**Live verified**:
```
recall@K: 1.000  (3/3 cases hit within top-5)
MRR:      0.778  (one case at rank 3, two at rank 1)
```

The rank-3 case (Kopfstand) is an actual signal: both CVs (16 EN, 25 DE) outrank the expected German one in the reranker's view. The kind of insight the harness exists to surface — addressable via reranker tuning, language-aware reranking, or richer chunk metadata.
